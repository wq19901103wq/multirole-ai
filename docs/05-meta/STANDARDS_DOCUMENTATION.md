# 文档守则

> 适用于：任何修改架构、接口、流程、配置或修复 bug 后需要更新文档的场景。

---

## 核心原则

**文档是代码的一部分**。修改代码而不同步更新文档，等同于提交不完整的工作。

---

## 必须同步更新文档的场景

| 变更类型 | 必须更新的文档 |
|---------|--------------|
| 新增/删除模块 | ARCHITECTURE.md |
| 接口签名变更 | API_SURFACE.md |
| 新增/修改文件职责 | MODULE_INDEX.md |
| 上手流程或配置变化 | AI_QUICKSTART.md |
| 踩到新坑 | LESSONS_LEARNED.md |

---

## 文档内部自洽

任何描述接口或架构的文档，自身必须逻辑一致：

1. **接口声明与调用示例一致**
2. **伪代码必须和文字描述一致**
3. **引用的函数/类必须在文档内有定义或链接**
4. **命名必须统一**

---

## 设计文档与实现代码的边界

| 属于设计文档的职责 | 不属于设计文档的职责 |
|-------------------|---------------------|
| 模块划分与依赖方向 | import 语句是否齐全 |
| 数据流与主循环逻辑 | 运行时状态的初始化 |
| 关键算法思路 | 标准库函数是否已 import |
| 接口签名与参数语义 | 详细的异常处理、日志输出 |

---

## 文档必须标注实现进度

强制使用以下标签之一：

| 标签 | 含义 |
|------|------|
| **未实现** | 仅设计阶段，代码不存在 |
| **开发中** | 代码已部分实现 |
| **开发完成** | 代码已实现，但未测试 |
| **测试完成** | 已通过测试，待合并 |
| **已上线** | 生产环境稳定运行 |

---

## 改文档前的强制检查

每次修改架构/接口文档时，必须：

1. **接口签名改了** - 全文搜索旧名称，确保示例全部更新
2. **新增/删除了类型/模块** - 检查层级依赖规则
3. **改了架构分层** - 检查 ASCII 图和依赖规则是否同步
4. **新增/删除了文件** - 同步检查 MODULE_INDEX.md
